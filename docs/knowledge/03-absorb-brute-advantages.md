Daniel, 2026-05-17 (ratified mid-loop, additive to [[feedback_atomic_action_principle]]
and [[project_atomic_ab_loop]]). The A/B loop's true objective: make the
Atomic OS **the best operating system on the market for CLI agents** at *real
complex software construction* — NOT "win every trivial benchmark".

**Why Normal still wins efficiency rounds:** brute advantages — less ceremony,
fewer tool calls, direct patch, direct autofix, starts writing sooner, no
trace, proves/preserves/policies less. **Atomic wins when it converts each
brute advantage into a superior atomic version, keeping its own strengths:**
- direct autofix → analyzer dry-run transaction
- direct patch → one batch atomic transaction
- manual refactor → macro-refactor operator
- free reading → AST / public-surface plan
- long prompt → fast-path compiler
- fixed rule → dynamic policy
- micro-edits → one macro-atomic intention

**Decisive lever for the residual efficiency loss:** Atomic only beats Normal
on duration/cost/tokens/turns when its **fast-path is as short as Normal's** —
fewest possible macro-atomic transactions, minimal preamble (no ToolSearch
flailing, no whole-file Read), reach the highest operator immediately, and a
measured "you are DONE — stop" so the model never does a second restructuring
wave. Operators alone aren't enough; the *number of round-trips to realize the
intention* must approach Normal's while trace/safety/preservation stay.

**Architecture to become the best CLI-agent OS (target the loop builds toward):**
1. Fixed kernel: security, path containment, protected files, trace, rollback,
   validation, no-façade.
2. Dynamic policy: the task defines operator, scope, validation, prompt,
   report, fast-path, budget.
3. Macro operators: lint-transaction, service-split, API-wiring, DB-migration,
   frontend↔backend wiring, webhook consumption, test-repair.
4. Benchmark loop: every defeat → diagnosis → operator/policy → tested on the
   same benchmark → escalate only on huge-margin win.
5. Cross-CLI: Codex, Claude Code, OpenCode (and future agents) obey one runtime.

**Honest scope:** some benchmarks favor Normal by design (one-line trivial
tasks, zero-risk, pure raw-speed, no trace/safety credit). The goal is not
"absolutely every benchmark" — it is **win everything that matters for real
complex tech construction**: correctness, preservation, cost, speed,
traceability, safety, continuity, multi-agent, non-technical trust. Under that
definition the potential is to make Atomic the *superior* mode, not the "safe"
mode. Condition: keep absorbing Normal's advantages without its defects. Loop
is infinite/continuous/uninterrupted per Daniel.

**PERMANENT MANIFESTO (Daniel, 2026-05-17 — atomicity = correct indivisibility,
NOT small size):** The atom is *the smallest unit of action that realizes the
intention without destroying preservable context* — at whatever level the
intention demands: char/space/comma · import · parameter · symbol · method ·
class · module · multi-file transaction · product flow. Atomic OS is NOT
"char-by-char development". Rule: **intention at the highest level possible;
action at the smallest level necessary; proof at the smallest sufficient
scope** — never a surface larger than the intention requires. Paradox: the
MORE mature the OS, the FEWER micro-edits it makes — it climbs to *larger but
still indivisible, intention-faithful* atoms (R-evidence: idempotency → class-
method extract → god-class auto-plan → TARGET-MET/STOP → LOC-sized plan →
extractability-aware plan = exactly grosseria→higher atomic operator).
Beating Normal is **metric, not mission**: Normal is only the baseline that
reveals where coarse action remains. Every Normal win → ask "which brute
advantage is not yet converted into a *better* atomic operation?" and convert
it (wrote-direct→atomic tx; started-sooner→atomic fast-path; manual-refactor
→macro-refactor operator; autofix→analyzer dry-run tx; knew-to-stop→TARGET-MET
/STOP). Never copy Normal's grosseria. Mission = expand atomicity across the
WHOLE CLI operational space (read · scope · plan · edit · refactor · validate ·
rollback · trace · prompt · shell · multi-agent · patch promotion · product
proof) so the agent stops being a *text generator* and becomes a *manipulator
of verifiable technological states*. Huge-margin victory over Normal is the
symptom that this atomicity became superior — not the goal itself.
